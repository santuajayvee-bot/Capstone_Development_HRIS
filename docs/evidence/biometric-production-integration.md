# LGSV HR Biometric Production Integration Evidence

## Panel Question 15

**Was the biometric system genuinely integrated or simulated?**

The production biometric attendance implementation is genuinely integrated with an actual **ZKTeco ZK9500 USB fingerprint scanner**. Attendance is not imported from a CSV or spreadsheet and is not generated by the production Node.js application.

The scanner is connected to an authorized Windows attendance station. A locally installed LGSV HR ZK9500 bridge uses the ZKTeco `libzkfpcsharp` SDK to acquire a fingerprint template from the scanner and match it against locally enrolled templates. After a successful match, the bridge sends attendance metadata to the production LGSV HR API over HTTPS.

The repository also contains separate simulation scripts for controlled integration tests and survey/UAT data. Those scripts are not part of the production attendance acquisition path. Production records created through the scanner are identified by the `BIOMETRIC_API` source and linked to a registered production device.

## Implemented Topology

```text
Employee finger
  -> ZKTeco ZK9500 USB fingerprint scanner
  -> ZKTeco libzkfpcsharp SDK
  -> LGSV HR Windows biometric bridge
     - captures the fingerprint
     - identifies the locally enrolled employee
     - applies a local duplicate-scan cooldown
  -> HTTPS POST https://lgsvhr.com/api/biometric/station-attendance
     with x-biometric-api-key
  -> Nginx reverse proxy on AWS EC2
  -> Node.js/Express biometric API
     - authenticates the registered device
     - validates the active employee and fingerprint mapping
     - rejects duplicates and invalid requests
     - determines the proper DTR punch
  -> Amazon RDS MySQL
     - biometric_scan_event
     - attendance_log
     - attendance_summary
     - system_audit_log
  -> Attendance Sync UI for HR validation
```

The fingerprint image and fingerprint template are not transmitted to AWS. Templates remain on the authorized Windows station. AWS receives only the employee identifier produced by the local match, registered device reference, scan time, and verification score.

## Synchronization Process

1. **Device initialization.** The Windows bridge initializes the ZKTeco SDK, opens the connected ZK9500, loads locally enrolled templates, and starts its background scanner.
2. **Fingerprint acquisition.** When a finger is placed on the ZK9500, the bridge calls the SDK's `AcquireFingerprint` operation.
3. **Local identification.** The bridge calls the SDK's `Identify` operation. An unmatched fingerprint is rejected locally and no attendance request is created.
4. **Payload creation.** For a successful match, the bridge creates an attendance payload containing `employee_id`, `device_id`, `scan_time`, and `verification_score`.
5. **Secure transmission.** The bridge sends the JSON payload through HTTPS to the AWS production endpoint and includes the device-specific API key in the `x-biometric-api-key` request header.
6. **Device authentication.** The Node.js API looks up the active `biometric_device` record and compares the supplied API key with the encrypted configured secret using a timing-safe comparison. Production does not permit unauthenticated device access.
7. **Server-side validation.** The API confirms that the employee exists, is active, and has an active mapping for the same registered device.
8. **Duplicate protection.** The bridge applies a local cooldown, while the server applies the configured duplicate window and a unique idempotency key. A duplicate request receives HTTP `409`.
9. **DTR classification.** The attendance policy engine classifies the scan into the appropriate AM/PM time-in or time-out slot.
10. **Transactional storage.** The API writes the scan event and attendance record in a database transaction, updates the attendance summary, appends an integrity entry, updates the device's last successful synchronization time, and writes an audit log.
11. **UI update and HR validation.** A real-time attendance event is emitted to the Attendance interface. The record remains subject to the configured HR validation workflow before it becomes payroll-ready.

There is no periodic file import in this production sequence. Each successful physical scan is synchronized as an individual HTTPS API transaction.

## Production Attendance API

### Endpoint

```http
POST https://lgsvhr.com/api/biometric/station-attendance
Content-Type: application/json
x-biometric-api-key: <device-specific secret>
```

### Request Body

```json
{
  "employee_id": 123,
  "device_id": "ZK9500-AWS-001",
  "scan_time": "2026-07-20T08:03:15+08:00",
  "verification_score": 87
}
```

The `employee_id` is the result of the local SDK fingerprint match. It is still verified against the active device-to-employee mapping on the server.

### Successful Response

```json
{
  "message": "TIME IN recorded from biometric scan.",
  "action": "TIME_IN",
  "attendance_id": 456,
  "employee": {
    "id": 123,
    "employee_code": "EMP00123",
    "name": "Employee Name"
  },
  "device_id": "ZK9500-AWS-001",
  "scan_time": "2026-07-20 08:03:15",
  "verification_score": 87,
  "result": "PENDING_VALIDATION"
}
```

### Relevant Error Responses

| HTTP status | Meaning |
|---:|---|
| `400` | Missing/invalid payload or attendance processing failure |
| `401` | Invalid biometric device API key |
| `403` | Inactive employee or fingerprint not mapped to the device |
| `404` | Device or employee not registered |
| `409` | Duplicate scan or duplicate DTR punch |
| `429` | Biometric/attendance API rate limit exceeded |

## Supporting Station APIs

| Method and endpoint | Purpose | Authentication |
|---|---|---|
| `POST /api/biometric/station-command/next` | The Windows bridge polls AWS for an enrollment, verification, or deletion command. | Device API key |
| `POST /api/biometric/station-command/:commandId/complete` | The bridge returns the result of a station command to AWS. | Device API key |
| `GET /api/biometric/station-status` | Displays the registered station and latest scan status. | Status-only endpoint |
| `GET /api/biometric/status` | Loads biometric health and latest scan evidence in the authenticated HR interface. | JWT and RBAC |
| `GET /api/attendance/biometric/health` | Provides protected device and synchronization health information. | JWT and HR/Admin RBAC |

The bridge also exposes a loopback-only listener, normally `http://localhost:8787`, for local health, enrollment, verification, deletion, and scanner operations. This listener does not replace the AWS production API.

## Data Stored in AWS

AWS stores:

- registered device metadata;
- encrypted biometric reference ID;
- SHA-256 biometric reference lookup hash;
- scan timestamp and verification score;
- DTR punch and validation status;
- payload and idempotency hashes;
- synchronization state; and
- security/audit logs.

AWS does **not** store:

- fingerprint images;
- raw fingerprint templates; or
- raw ZKTeco SDK capture buffers.

## Short Defense Answer

> The production biometric flow is genuinely integrated, not simulated. We use an actual ZKTeco ZK9500 USB scanner and its `libzkfpcsharp` SDK through a Windows bridge. The bridge captures and matches the fingerprint locally, then sends only attendance metadata over HTTPS to `https://lgsvhr.com/api/biometric/station-attendance` using a registered device API key. AWS validates the device, employee, mapping, timestamp, and duplicate rules before saving the event to RDS MySQL and displaying it in Attendance Sync. Fingerprint images and templates remain on the attendance station and are not uploaded to AWS. Separate simulation scripts exist only for controlled tests and are not used by the production scanner path.
