# Performance Management Module - STRIDE Evidence

| STRIDE | Possible threat | Security control | Implementation evidence | How to show it |
|---|---|---|---|---|
| Spoofing | Another user impersonates the official evaluator or an unattended HR session finalizes a rating. | JWT session authentication, database-backed raw-role verification, employee ownership checks, and Argon2id step-up password re-authentication for finalization and reopen. | `server/performance-management.js` applies `requireAuth`, accepts evaluator actions only when the database role is exactly `hr_manager`, and calls the shared `verifyPassword` service before finalization/reopen. | Login as HR Manager, open Performance, finalize a review, and show the current-password confirmation. Call finalize without `currentPassword` and show `401`. |
| Tampering | Indicator ratings, HR remarks, recommendations, goals, or workflow fields are changed directly in the request or database. | Request allowlists, strict integer 1-4 rating validation, parameterized SQL, optimistic version checks, locked finalized records, SHA-256 integrity hash, and visible integrity mismatch state. | The API accepts only the five approved criteria and twenty approved indicators, encrypts the rating set, requires `version`, locks finalized reviews, and computes `integrity_hash` over encrypted ratings plus narratives. GET verification returns `MISMATCH` when finalized data differs. | Finalize a test review, change the encrypted indicator-rating value directly in the test database without updating `integrity_hash`, refresh Performance, and show the red Mismatch row. Undo the change after the demonstration. |
| Repudiation | HR Manager denies assigning, updating, finalizing, or reopening an evaluation. | Timestamped audit trail with actor, target employee, IP address, user agent, action, and non-sensitive workflow metadata. | Every workflow write records a `PERFORMANCE` entry in `system_audit_log`; reopen requires a reason and fresh step-up verification. | Complete one workflow action, then open System Administration > Audit Trail, filter module `PERFORMANCE`, and show the actor and timestamp. |
| Information Disclosure | An employee reads another employee's evaluation or sees an unfinished rating. | Backend RBAC, employee self-scope/IDOR checks, finalized-only employee queries, AES-256-GCM encryption of indicator ratings, goals, and HR narratives at rest, and no-store API responses. | HR Manager-only routes use `requirePerformanceManager`; employee list/detail queries require the authenticated `Employee_ID` and `FINALIZED` status; evaluation fields are decrypted only after access checks. | Login as Regular Employee and request another or unfinished review ID in Postman; show `403`. Inspect the database evaluation columns and show ciphertext rather than readable ratings or remarks. |
| Denial of Service | Repeated list/write/finalize requests exhaust the API or database pool. | Dedicated audited Performance API rate limiter plus the shared API limiter, bounded list queries, and transaction scope limited to state changes. | `server.js` applies `PERFORMANCE_ROUTE_RATE_LIMIT` to `/api/performance`; list queries use fixed maximum limits; rate-limit blocks are audited under `PERFORMANCE_SECURITY`. | Send repeated requests to `/api/performance/overview` and show `429 Too Many Requests` with `Retry-After`. |
| Elevation of Privilege | Employee, HR Admin, System Administrator, or payroll staff attempts to create or alter an evaluation. | Exact database-role guards on every management endpoint; client-side visibility is not treated as authorization. | Cycle creation/status, assignment, evaluation, finalization, reopen, and eligible-employee endpoints require the raw `hr_manager` role. Regular Employees only receive their own finalized records. | Call `POST /api/performance/cycles` as an employee, System Administrator, or payroll role and show `403 Access denied`. |

## Workflow Evidence

`DRAFT cycle -> ACTIVE cycle -> ASSIGNED -> FINALIZED`

Only the HR Manager evaluates and finalizes. Regular Employees cannot rate, edit, delete, or acknowledge evaluations; they can only view their own finalized result. Finalization locks the review and creates its integrity hash. A correction requires the HR Manager to reopen it with a reason and current-password step-up, which clears the old finalization proof and creates an audit entry before editing can resume.

Each evaluation uses five criteria with four observable indicators each. Every indicator accepts only an integer rating from 1 to 4. The system averages the four indicators for each criterion, then averages the five criterion results for the overall score. A review cannot be finalized while any indicator is unrated.
