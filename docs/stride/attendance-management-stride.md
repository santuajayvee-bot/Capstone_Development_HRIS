# Attendance Management Module - STRIDE Evidence

| STRIDE | Possible threat | Security control | Implementation evidence | How to show it |
|---|---|---|---|---|
| Spoofing | Fake attendance submission, use of another account, or use of an unattended HR session to validate/reject time-in/time-out records. | Authentication, verified user identity, biometric device authentication, and step-up password re-authentication before HR validation/rejection. | `server/attendance.js` uses `requireAuth` and `requireRole`; employee view is scoped to `/api/attendance/my-records`; biometric station/device routes are protected; `PATCH /api/attendance/:id/verify` requires `currentPassword` for `PAYROLL_READY` and `REJECTED`, verified with Argon2id. | Login as HR Admin and click Validate/Reject attendance; show the password confirmation modal. Try the API without `currentPassword` and show `401`. |
| Tampering | Time-in/time-out records are modified or validation status is changed without authorization. | Server-side validation, HR-only correction routes, reason requirement, step-up password verification for validate/reject, audit logs, integrity chain. | `PATCH /api/attendance/:id/override`, `verify`, `overtime`, and `overtime-review` require HR roles, validate fields, write `attendance_adjustment`, append integrity entries, and log to `system_audit_log`. | Login as employee and try correction endpoint: `403`. Login as HR Admin, validate/reject with password, then show Attendance Audit Log or System Admin Audit Trail. |
| Repudiation | Employee or HR denies attendance entry/change. | Timestamped attendance logs and audit trail. | `system_audit_log`, `attendance_adjustment`, and `attendance_integrity_chain` record manual attendance, corrections, verification, overtime updates, actor, timestamp, old/new values, and reason. | Open Attendance > Audit Log and show the correction/manual/verification entry. |
| Information Disclosure | Attendance reports are exposed. | RBAC and restricted reports/record access. | Employees use `/api/attendance/my-records`; HR/Payroll/Admin roles use `/api/attendance/all`, `/summaries`, `/audit-log`; frontend hides restricted tabs and backend enforces roles. | Login as employee and show no Audit/Biometrics management tab. Use Postman to access `/api/attendance/audit-log` as employee and show `403`. |
| Denial of Service | Attendance or biometric service becomes unavailable from repeated requests or scanner-command flooding. | Dedicated attendance/biometric rate limiting, general API rate limiting, biometric command single-flight, retry handling, and health/status monitoring. | `server.js` applies `ATTENDANCE_RATE_LIMIT` to `/api/attendance` and `/api/biometric`; `server/biometric.js` rejects concurrent station commands; `public/js/attendance.js` honors `429 Retry-After`; `/api/attendance/biometric/health` provides protected status evidence; `/health` and PM2 show service health. | Send repeated requests to `/api/attendance/...` or `/api/biometric/...` and show `429`. Start two biometric commands and show busy response. Open Attendance > Biometrics to show health/status indicators. |
| Elevation of Privilege | Employee edits attendance records directly. | Restricted edit permissions and backend RBAC. | Manual attendance, correction, verification, overtime, audit, and biometric management routes require HR/Admin roles; employee UI cannot access those controls and backend denies direct calls. | Login as Regular Employee and attempt `PATCH /api/attendance/:id/override`: `403 Forbidden`. |

## DoS Evidence Statement

The Denial of Service control is implemented, not partial. The module has a dedicated Attendance/biometric rate limiter, a broader API limiter, single-flight protection for biometric station commands, `429 Retry-After` handling in the Attendance UI, and protected health/status monitoring for biometric synchronization.
