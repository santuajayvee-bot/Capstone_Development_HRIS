BLOCKCHAIN MODULE DEFENSE NOTES
LGSV HR System - System Admin / Blockchain Integrity Module

Use this as your speaking guide during defense.


1. OPENING EXPLANATION

Good day. I will explain the Blockchain Integrity Module of our LGSV HR system.

This module is mainly handled under the System Admin side because it is part of system audit, integrity verification, and security monitoring.

The purpose of this module is to check whether finalized payroll or finalized attendance DTR records were changed after approval.

It is important to clarify that blockchain is not our main database. Our main database is still MySQL. Employee information, payroll details, attendance records, payslip data, biometric references, and other sensitive HR data are stored off-chain in MySQL.

The blockchain is used only as a tamper-evidence layer.

In simple terms, this module answers the question:

"After payroll or DTR was finalized, was the record changed or tampered with?"


2. WHY WE USE BLOCKCHAIN

We use blockchain because payroll records must be reliable and auditable.

In HR and payroll systems, one major risk is unauthorized modification after approval. For example, someone might try to change net pay, deductions, or attendance totals after payroll has already been finalized.

To detect that, the system creates a hash of the finalized record and records that proof to Hyperledger Fabric.

Hyperledger Fabric is a permissioned blockchain. This means it is not public like Bitcoin or Ethereum. It is designed for private organizations where only authorized participants can access the network.

This is appropriate for an HR system because payroll and employee data are confidential.


3. WHAT DATA IS STORED ON-CHAIN

We do not store full personal information on blockchain.

The blockchain stores only proof and metadata, such as:

- Payroll or DTR reference ID
- SHA-256 hash
- Approval metadata
- Transaction reference
- Verification result

Full employee PII, names, salary details, attendance logs, biometrics, and payslip data remain off-chain in MySQL.

This design protects privacy while still allowing integrity verification.


4. SIMPLE HASH EXPLANATION

A hash is like a digital fingerprint of a record.

If the record stays the same, the hash stays the same.

If even one value changes, the hash becomes different.

Example:

Original finalized net pay:
10000.00

If someone changes it to:
10001.00

The new hash will not match the blockchain hash anymore.

That is how the system detects possible tampering.


5. PAYROLL BLOCKCHAIN PROCESS

For payroll, the process starts after payroll is finalized.

Step 1: Payroll Manager approves and finalizes payroll.

Step 2: The system gets the important finalized payroll fields from MySQL.

Examples of fields included in the payroll hash:

- Payroll_ID
- Employee_ID or employee reference
- Gross_Pay
- Total_Statutory_Deductions
- Net_Pay
- Non_Taxable_Allowance
- Approval_Status
- Finalized_At
- Approved_By

Step 3: The system creates a SHA-256 hash from those finalized fields.

Step 4: The system records the hash and metadata to Hyperledger Fabric.

Step 5: Fabric returns a transaction reference.

Step 6: The transaction reference is saved back in MySQL so the MySQL payroll record is linked to the blockchain proof.


6. ATTENDANCE DTR BLOCKCHAIN PROCESS

For attendance DTR, the same concept is used.

The system only records finalized DTR proofs.

Examples of fields included in the DTR hash:

- DTR_ID
- Employee reference
- Date range start
- Date range end
- Total work hours
- Late minutes
- Undertime minutes
- Overtime hours
- Attendance status
- Generated by
- Verified by
- Finalized date

The system creates a SHA-256 hash from these finalized DTR fields and anchors that proof to Hyperledger Fabric.


7. HOW SYSTEM ADMIN VERIFIES INTEGRITY

The System Admin does not manually compare MySQL records and blockchain records by reading them line by line.

The System Admin clicks the Verify Integrity button.

After clicking that button, the backend performs the actual comparison.

The system does this:

1. Reads the current finalized payroll or DTR record from MySQL.
2. Rebuilds the same hash payload.
3. Computes a new SHA-256 hash.
4. Gets the stored hash from Hyperledger Fabric.
5. Compares the current MySQL-computed hash with the Fabric hash.
6. Returns the verification result.

If the hashes match, the record is marked as Verified.

If the hashes do not match, the system marks it as Mismatch Detected or Critical.

This means the off-chain MySQL data may have been changed after finalization.


8. SIMPLE COMPARISON EXAMPLE

Case 1: No tampering

MySQL computed hash:
abc123

Fabric stored hash:
abc123

Result:
Verified

Meaning:
The current MySQL data still matches the blockchain proof.


Case 2: Possible tampering

MySQL computed hash:
xyz789

Fabric stored hash:
abc123

Result:
Mismatch Detected

Meaning:
The current MySQL data no longer matches the blockchain proof.


9. STATUS EXPLANATION IN THE MODULE

Waiting for Approval
- The payroll or DTR is not final yet.
- It should not be recorded to blockchain yet.

Waiting for Fabric
- The record is finalized locally in MySQL.
- The system is waiting to anchor it to Hyperledger Fabric.
- This can happen if Fabric is temporarily unavailable or the network is not ready.

Recorded on Fabric
- The finalized hash has been successfully recorded to Hyperledger Fabric.
- A Fabric transaction reference exists.

Verified
- The current MySQL record matches the blockchain proof.
- This means no mismatch was detected.

Mismatch Detected
- The current MySQL record does not match the blockchain proof.
- This may indicate tampering or unauthorized modification.

Failed
- A technical error happened during anchoring or verification.
- It needs review or retry.


10. ROLE SEPARATION

We separated responsibilities by role.

Payroll Manager:
- Approves and finalizes payroll.
- Can view payroll blockchain evidence.
- Should not be the final technical verifier.

System Admin:
- Performs integrity verification.
- Can verify recorded hashes against Hyperledger Fabric.
- Can inspect audit trails and technical integrity results.

Payroll Officer:
- Can view payroll integrity status for audit awareness.
- Cannot verify integrity.

HR Admin / HR Manager:
- Can view attendance DTR integrity evidence depending on scope.
- They are not the final technical verifier.

Regular Employee:
- Should not access the full Blockchain Integrity module.
- At most, they can see the integrity status of their own payslip if shown in employee self-service.

The reason for this role separation is audit control.

The same role that approves payroll should not also be the final technical verifier.

Simple explanation:

Payroll Manager approves.
System Admin verifies.
The system performs the hash comparison.


11. AUDIT TRAIL EXPLANATION

Every important blockchain action creates an audit trail.

The audit trail records:

- Event type
- Status
- Actor role
- Timestamp
- Payload hash
- Transaction hash

Example event:

VERIFY_INTEGRITY

If the verification succeeds, the status becomes:
VERIFIED

If the hashes do not match, the status becomes:
CRITICAL

This gives accountability because the system records who triggered the verification, when it happened, and what the result was.

13. SHORT DEFENSE ANSWER

If the panel asks:
"How exactly does your blockchain verification work?"

Answer:

"The System Admin triggers the verification. The backend reads the current finalized MySQL payroll or DTR record, rebuilds the hash payload, computes a SHA-256 hash, then compares that hash with the hash stored in Hyperledger Fabric. If both hashes match, the record is verified. If they do not match, the system flags the record as a possible tampering case and records the result in the audit trail."


14. IF THEY ASK WHY PAYROLL MANAGER CANNOT VERIFY

Answer:

"The Payroll Manager is the business approver of payroll. To preserve separation of duties, the technical integrity verification is assigned to System Admin. This prevents the same role from both approving payroll and validating the integrity result."


15. IF THEY ASK WHY FULL DATA IS NOT ON BLOCKCHAIN

Answer:

"We do not store full employee or payroll data on-chain because HR data is sensitive. Instead, full data stays off-chain in MySQL, and only hashes and references are stored on Hyperledger Fabric. This gives integrity verification without exposing personal information."


16. IF THEY ASK WHAT HAPPENS IF FABRIC IS DOWN

Answer:

"If Fabric is temporarily unavailable, the system does not lose the finalized record. It marks it as Waiting for Fabric or Pending Anchor. This means the record is finalized locally and waiting to be anchored once Fabric is available. The audit trail still records the attempt."


17. IF THEY ASK WHAT MISMATCH MEANS

Answer:

"A mismatch means the current MySQL data produced a different hash from the hash stored in Fabric. Since hashes change when record values change, this indicates that the off-chain data may have been modified after finalization."


18. CLOSING STATEMENT

To summarize, the Blockchain Integrity module is a tamper-evidence layer for finalized payroll and attendance DTR records.

It does not replace MySQL and it does not store full HR data on-chain.

The system stores full data off-chain, creates SHA-256 hashes after finalization, records the proof to Hyperledger Fabric, and allows the System Admin to verify whether the current MySQL data still matches the blockchain proof.

This improves payroll accuracy, auditability, and detection of unauthorized changes after final approval.
